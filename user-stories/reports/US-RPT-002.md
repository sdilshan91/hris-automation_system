# US-RPT-002: Leave and Attendance Reports

## 1. Description
**As an** HR Officer,
**I want to** generate reports on leave utilization, leave balances, attendance summaries, and absenteeism trends with filters for period, department, and employee,
**So that** I can monitor workforce availability, identify attendance issues, ensure leave policy compliance, and plan resource allocation effectively.

## 2. Preconditions
- The HR Officer is authenticated and has `Reports.View.All` permission within their tenant.
- Leave and attendance data exists in the tenant (Leave and Attendance modules are populated).
- The Reporting module is enabled for the tenant's subscription plan.

## 3. Acceptance Criteria (IEEE 830 S3.2 - Specific Requirements)
| # | Given | When | Then |
|---|-------|------|------|
| AC-1 | HR Officer selects "Leave Utilization" report for Q1 2026 | They generate the report | The report shows: total leave days taken by type (bar chart), average leave utilization per department (grouped bar chart), employees with highest leave consumption (top-10 table), and leave type distribution (donut chart). All data is tenant-scoped. |
| AC-2 | HR Officer selects "Leave Balance" report as of today | They generate the report | A table is displayed showing each employee's current balance per leave type (earned, sick, casual, etc.), with color-coded indicators: green (> 50% remaining), yellow (25-50%), red (< 25%). |
| AC-3 | HR Officer selects "Attendance Summary" for the current month | They generate the report | The report shows: total working days, average attendance percentage, late arrivals count, early departures count, overtime hours (bar chart by department), and absenteeism rate per department (line chart). |
| AC-4 | A Manager selects "Leave Utilization" for their team | They generate the report | Only their direct reports' data is included in the report (permission-based scoping via `Reports.View.Team`). The same report generated by HR includes all employees. |
| AC-5 | Leave reports are generated in Tenant A and Tenant B | Each tenant views their report | Each report shows only their own tenant's data; RLS and EF Core filters prevent cross-tenant data access. |

## 4. Functional Requirements (IEEE 830 S3.2)
- FR-1: The system SHALL provide the following leave and attendance reports: Leave Utilization, Leave Balance Summary, Attendance Summary, Absenteeism Trends, Overtime Report, Late Arrival Report.
- FR-2: Each report SHALL support filters: date range, department (multi-select), location, employee (search), leave type, shift.
- FR-3: Leave Utilization SHALL show consumption vs. entitlement per leave type with percentage utilization.
- FR-4: Attendance Summary SHALL calculate: attendance rate = (present days / working days) x 100, absenteeism rate = (absent days / working days) x 100.
- FR-5: Reports SHALL render charts using Chart.js / ngx-charts and provide a data table toggle.
- FR-6: The system SHALL support drill-down from aggregate data to individual employee details (click on a department bar to see employees).
- FR-7: Reports SHALL cache results in Redis (TTL 5 minutes) keyed by tenant, report type, and filter hash.
- FR-8: The system SHALL scope data based on the user's permission: `Reports.View.All` for full tenant, `Reports.View.Team` for direct reports only.

## 5. Non-Functional Requirements (IEEE 830 S3.3)
- NFR-1: Report generation SHALL complete within 3 seconds (P95) for tenants with up to 5,000 employees.
- NFR-2: All report data SHALL be isolated by tenant via PostgreSQL RLS and EF Core global query filters.
- NFR-3: Charts SHALL render within 1 second on the client side.
- NFR-4: The report UI SHALL be fully responsive from 360px to 4K resolution.
- NFR-5: The report UI SHALL meet WCAG 2.1 AA accessibility standards.
- NFR-6: Complex attendance calculations SHALL use PostgreSQL views for query optimization.

## 6. Business Rules
- BR-1: Leave balances are calculated as: entitlement + carry forward - consumed - pending approval.
- BR-2: Working days are determined by the tenant's working calendar (excluding public holidays and weekly offs configured per shift).
- BR-3: Overtime is calculated based on attendance logs exceeding the shift's standard hours.
- BR-4: Absenteeism excludes approved leave; only unauthorized absences are counted.
- BR-5: Reports respect the tenant's leave year start (configurable: calendar year or custom fiscal year).
- BR-6: Data for terminated employees is included in historical reports but excluded from current balance reports.

## 7. Data Requirements
**Filter inputs:**
| Field | Type | Required | Validation |
|-------|------|----------|------------|
| report_type | varchar(50) | Yes | Valid leave/attendance report key |
| date_from | date | No | Defaults to start of current month |
| date_to | date | No | Defaults to today |
| department_ids | uuid[] | No | Must exist in tenant |
| employee_id | uuid | No | Must exist in tenant |
| leave_type_ids | uuid[] | No | Must exist in tenant |
| shift_ids | uuid[] | No | Must exist in tenant |

**Output:** Report object with chart data and tabular data, scoped by permissions.

## 8. UI/UX Notes
- Report selection: tabbed navigation within the "Leave & Attendance Reports" section (Leave Utilization | Leave Balance | Attendance Summary | Overtime | Absenteeism).
- Leave Balance table: use a heat-map style coloring for balance cells (green to red gradient).
- Attendance summary: use a calendar heat-map visualization showing daily attendance status per employee (small colored squares for each day).
- Drill-down: click on a chart segment to see individual records in a slide-out panel.
- On mobile (< 768px): charts stack vertically; tables become horizontally scrollable with sticky first column (employee name).
- Print button for generating a printer-friendly version.

## 9. Dependencies
- Leave Management module: Leave data (requests, balances, types).
- Attendance module: Clock-in/out logs, shift configurations.
- US-CHR-001: Employee records for employee context.
- US-CHR-004: Department data for grouping.
- US-RPT-004: Export functionality.
- Authentication module: Permission-based data scoping.

## 10. Assumptions & Constraints
- Leave balances are pre-computed and stored (or computed via PostgreSQL views) for report performance.
- Attendance calculations depend on shift configurations being correctly set up by the Tenant Admin.
- Chart.js or ngx-charts (free/open-source) is used for visualizations.
- Only free/open-source libraries are used.
- The system uses PostgreSQL with RLS as defense-in-depth for tenant isolation.

## 11. Test Hints
- **Leave Utilization:** Create 10 employees with varied leave consumption; generate report; verify totals and chart data match.
- **Leave Balance:** Verify balance calculation: entitlement 20, consumed 8, pending 2 = balance 10.
- **Attendance Summary:** Create attendance logs for 20 working days; 18 present, 2 absent; verify attendance rate = 90%.
- **Manager scope:** Log in as Manager; verify report only includes direct reports.
- **Tenant isolation:** Generate the same report in Tenant A and B; verify data independence.
- **Overtime:** Log hours exceeding shift standard for 3 employees; verify overtime report shows correct excess hours.
- **Calendar heat-map:** Verify the heat-map correctly colors days (green = present, red = absent, grey = holiday).
- **Responsive:** Test at 360px width; verify chart stacking and table scrolling.
