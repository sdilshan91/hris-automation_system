Attendance test cases (`test-cases/attendance/`) follow the same conventions as Leave Management (see [[project-leave-tc-conventions]]), established by US-ATT-001 (first story; created the dir + TEST-MATRIX):

- **Two parallel ID sequences:** functional/security/perf/a11y TCs use a single running `TC-ATT-NNN` counter across all stories (do NOT restart per story); dedicated multi-tenant isolation TCs use a separate `TC-ATT-ISO-NNN` counter. Each story gets exactly 4 ISO TCs (cross-tenant read visibility, no/invalid-tenant-context rejection, cross-tenant write/data-layer block, tenant-scoped cache keys). US-ATT-001 used TC-ATT-001..012 + TC-ATT-ISO-001..004.
- **Three artifacts per story:** per-TC `.md` files, `test-cases/attendance/TEST-MATRIX.md`, and the root `test-cases/TRACEABILITY-MATRIX.md` (forward + backward tables, per-story Detailed Requirements Traceability, per-story Coverage Summary, the Cross-Module table totals, and the closing note paragraph).
- **Tenant isolation mechanism:** this codebase enforces isolation via EF Core global query filters + TenantInterceptor, NOT Postgres RLS. US-ATT-001 NFR-2/S10 say "RLS" — ISO TCs describe the EF mechanism and note RLS as an extension point. Same caveat as leave-management per vault.
- **Redis cache (FR-6):** not assumed wired. Write cache-dependent steps as CONDITIONAL with a DB-fallback verification path (mirrors leave-management deferred-Redis handling, see [[project-leave-deferred-infra]]).

**Why:** Reviewers and the orchestrator rely on consistent IDs and the three-matrix structure for backward traceability; breaking the sequence or skipping a matrix breaks links.

**How to apply:** Before writing the next Attendance story's TCs, glob existing `TC-ATT-*` files for the highest functional and highest ISO numbers and continue both sequences. BR-6 selfie-photo (require_photo) was reported to the caller as having no AC under US-ATT-001 — revisit if a photo-capture story appears.

**ISO-TC reuse for derived operations (US-ATT-002 precedent):** A story that operates on the SAME table US-ATT-001 already covered (clock-out mutates the same `attendance_log`) does NOT need 4 fresh ISO TCs. US-ATT-002 added only ONE new ISO TC — TC-ATT-ISO-005 (Tenant A employee cannot clock out Tenant B's open record, the distinct write mutation) — and explicitly REUSED TC-ATT-ISO-001..004 for table-level read/missing-context/cache isolation, referencing them in both the TC body and the matrices. The mandatory "every module gets isolation tests" gate is satisfied by the existing 4 plus the operation-specific one. US-ATT-002 = TC-ATT-013..024 + TC-ATT-ISO-005. The auto-clock-out Hangfire job TC (TC-ATT-021) also carries a tenant-scoping step, so the job's isolation is covered there rather than in a separate ISO TC.

**US-ATT-004 (manager approve/reject regularization) precedent:** = TC-ATT-037..050 (14 TCs) + ONE new ISO TC (TC-ATT-ISO-007), reusing ISO-001..004 AND ISO-006. ISO-007 covers the approve/reject MUTATION isolation (queue invisible, 404 on fetch, no cross-tenant approve/reject/bulk, no Tenant B attendance_log write). NEW deferral seams written as CONDITIONAL (not gaps), all consistent with prior ATT stories + leave: AC-4/FR-4/BR-4 multi-level workflow DEFERRED on Approval Workflow Engine (US-ADM-007) -- single-level final-approval (TC-ATT-037) + deny-self-approval (TC-ATT-042) verified live (same handling as US-LV-005 TC-LV-097); FR-5 employee notification on approve/reject DEFERRED on US-NTF (dispatch seam incl. rejection-reason-in-payload verified now); FR-8 Redis status cache CONDITIONAL on Redis; BR-5 payroll-lock-AT-APPROVAL CONDITIONAL on Payroll (complements US-ATT-003 TC-ATT-029 submit-time lock). Exact AC-5 string quoted verbatim: "You are not authorized to approve requests for this employee." attendance_log create-on-approval (FR-2) is the approval-side counterpart to US-ATT-003's submit-side no-log deferral. Module now 4 stories / 57 TCs / 7 ISO.

**US-ATT-005 (shift management) precedent:** = TC-ATT-051..066 (16 functional/security/perf/a11y) + ONE new ISO TC (TC-ATT-ISO-008), reusing ISO-001..004 for context/cache. First Attendance story to introduce NEW tables (`shift`, `employee_shift`); the single dedicated ISO-008 covers list/fetch/edit/delete/clone/assign/resolve isolation + body-injected tenant_id/employee_id + subdomain-switch for both tables. Two performance TCs this story (TC-ATT-064 pages <2s NFR-1; TC-ATT-065 bulk-assign-500 <5s NFR-2). Exact delete-when-assigned string quoted verbatim: "This shift is assigned to {N} employees. Please reassign them before deleting." (409 code `shift_in_use`, {N} dynamic). NEW deferral seams written CONDITIONAL (not gaps), consistent w/ prior ATT + leave: NFR-4 Redis shift cache DEFERRED (DB-fallback in TC-ATT-064, cache-key isolation reuses ISO-004); NFR-3 RLS-vs-EF-query-filter extension point (TC-ATT-ISO-008); BR-1 tenant default-shift PROVISIONING auto-seed DEFERRED on Tenant Admin (TC-ATT-058 verifies fallback vs manually-flagged default); BR-4 late-arrival flagging from grace_period DEFERRED on US-ATT-008 (TC-ATT-062 verifies shift-def side, threshold exposed); S10 night-shift end<start is VALID not BR-7-zero-duration (TC-ATT-055 def-side cross-midnight; end-to-end clock calc owned by US-ATT-001/002); BR-5 break auto-deduct owned by US-ATT-002 (TC-ATT-018). FLEXIBLE shift only validates minimum_hours, start/end optional (BR-8, TC-ATT-054). Module now 5 stories / 74 TCs / 8 ISO.

**US-ATT-003 (regularization) precedent:** = TC-ATT-025..036 + ONE new ISO TC (TC-ATT-ISO-006), reusing ISO-001..004 — even though regularization introduces a NEW table (`attendance_regularization`), the single dedicated ISO-006 covers read + submit + cross-tenant link/approver targeting for that table, and ISO-001..004 still cover the cross-cutting context/cache mechanism. Two NEW deferral seams beyond Redis/RLS were introduced here and written as CONDITIONAL (not gaps): FR-4 manager in-app notification is DEFERRED on the Notification System (US-NTF, not built) — verified the dispatch SEAM (recipient = line manager, tenant-scoped, payload refs regularization_id) now; FR-7/BR-6 payroll-period lock is CONDITIONAL on the Payroll module — verified the unlocked path + exact error-contract now. FR-3 workflow: assert only that a workflow_instance is initiated on submit; approve/reject side belongs to US-ATT-004. Exact AC error strings must be quoted verbatim in the TC ("...the last {N} days.", "A pending regularization request already exists for this date.", "This date falls within a locked payroll period. Please contact HR.").
